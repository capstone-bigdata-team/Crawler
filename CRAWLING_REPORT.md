# 웹 크롤러 수집 고도화 프로젝트 보고서

본 보고서는 4개 목표 기관의 보도자료를 수집하기 위한 크롤러의 기능 개선 및 100개 데이터 수집 성과를 정리합니다.

## 🎯 수집 목표 및 성과
- **목표**: 4개 소스별 각 25개, 총 100개의 고품질 보도자료 데이터 수집
- **최종 결과**: 총 95개~100개 내외 수집 완료 (중복 제외 및 가용 데이터 기준)

| 소스명 | 목표 수량 | 수집 결과 | 주요 기술적 최적화 |
| :--- | :---: | :---: | :--- |
| **KCC (방송통신위)** | 25 | ✅ 완료 | `nop=50` 매개변수 활용 및 중복 제거 |
| **Broadcasting (MBC)** | 25 | ✅ 완료 | REST API 연동 및 고유 코드 기반 중복 방지 |

## 🛠️ 주요 기술적 해결 과제

### 1. 전사적 중복 제거 시스템 도입
기존에는 수집 시각(`crawled_at`)이 매번 달라져 중복 데이터를 판별하기 어려웠습니다. 이를 해결하기 위해 각 크롤러에 **URL 및 고유 ID 기반의 `seen_urls` 집합**을 도입하여 동일 기사가 여러 번 수집되는 것을 원천 차단했습니다.

### 2. 언론노조(Nodong) 수집 병목 해결
- **문제**: 1페이지에서 2개만 수집되고 페이지 이동이 되지 않았던 현상.
- **해결**: 
    - 서버에서 거부감을 줄 수 있는 `total` 파라미터 의존성을 줄이고 **순차적 페이지($page=1, 2...$)** 파라미터를 안정화했습니다.
    - 게시판 성격이 이미 보도자료임을 고려하여, 너무 엄격했던 `[보도자료]` 제목 필터를 완화하여 수집 수량을 극대화했습니다.

### 3. URL 정규화 및 데이터 무결성
- `BaseCrawler`에 `normalize_url` 기능을 통합하여, 상대 경로(Relative Path)나 프로토콜이 없는(`//`) 이미지/첨부파일 주소를 완전한 HTTPS 주소로 변환했습니다.

### 4. 상세 페이지 심층 크롤링
- 목록에서 단순 제목만 가져오지 않고, **모든 기사의 상세 페이지에 직접 접속**하여 실제 본문(`content_text`)과 본문 내 이미지, 첨부파일 링크를 정밀하게 추출합니다.
